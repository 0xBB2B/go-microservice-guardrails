# 仓库结构与模块落位

## 主题目标

用稳定的多模块工作区结构承载 Go 微服务，保证契约、共享基础设施、服务实现和联调资源各归其位。

## 必须遵守

- 根目录负责 `go.work`、总 README、规范文档和统一构建上下文，不承载某个业务服务源码
- 共享契约真源只放在 `contracts/`
- 共享基础设施能力只放在 `pkg/`
- 服务实现统一放在 `services/<service>/`
- 跨服务联调、部署和 E2E 资源统一放在 `ops/`
- 服务私有配置统一放在 `services/<service>/internal/config`
- 新增可独立构建的 Go 子模块时，显式加入 `go.work`

## 常见错误

- 在根目录直接放某个服务源码
- 在服务目录中维护另一份共享 proto 真源
- 把领域逻辑、服务私有配置或服务私有鉴权流程放进 `pkg/`
- 把服务私有业务代码放进 `ops/`
- 同一套联调入口在服务目录和 `ops/` 中重复维护

## 落地检查点

- 新增服务时，目录是否落在 `services/<service>/`
- 新增共享能力时，是否真的无业务语义且可跨服务稳定复用
- 新增联调或部署资源时，是否统一进入 `ops/`
- 任何新增 Go 模块后，`go.work` 是否同步更新
- 服务配置目录是否收敛到 `internal/config`
