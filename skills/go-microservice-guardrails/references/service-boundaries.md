# 服务边界与接口所有权

## 主题目标

确保服务通过公开契约协作，而不是通过复用彼此内部实现来“假装解耦”。

## 必须遵守

- 每个服务对自己的业务逻辑、数据模型、配置模型和运行语义负责
- 服务之间通过公开契约协作，不直接依赖对方内部函数、内部数据结构、仓储实现或私有配置
- 接口字段语义归被调用方拥有
- 当代码复用与边界清晰冲突时，优先保留服务边界
- 共享模块只承载无业务语义的公共能力

## 常见错误

- 调用方替被调用方定义接口语义
- 直接 import 其他服务的 `internal` 代码或仓储层
- 为了去重把领域逻辑提到共享模块
- 把“同一个团队维护”误当成可以跨边界复用内部实现的理由

## 落地检查点

- 新接口的所有权是否归被调用方
- 服务协作是否完全通过契约、SDK 或治理过的事件结构进行
- 任何共享代码是否都不携带领域语义
- 重构去重后，服务是否仍可独立理解、独立测试、独立发布
