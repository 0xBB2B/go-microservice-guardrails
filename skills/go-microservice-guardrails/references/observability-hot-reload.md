# 可观测性与热更新

## 主题目标

让所有会影响请求结果的动态状态都具备可观测证据、受控刷新路径和失败兜底策略。

## 必须遵守

- 关键请求具备可追踪能力，入口创建根级追踪上下文并向下游透传
- 日志与 Trace 至少有一个稳定关联键
- 影响用户结果、内部调用结果或状态切换结果的错误必须留证
- 配置、密钥、策略、缓存等动态状态的刷新、切换和失败行为必须可见
- 热更新要明确触发方式、生效时延、失败回退和并存窗口策略
- 刷新或切换尽量原子完成，避免长时间半新半旧

## 常见错误

- 刷新失败但没有日志、指标或 Trace 事件
- 日志与 Trace 互相对不上
- 动态状态切换后无法确认是否生效
- 只有刷新路径，没有失败回退或兜底策略
- 缓存命中、刷新、回退行为不可观测

## 落地检查点

- 关键链路是否统一透传 Trace 上下文
- 结构化日志是否带有稳定请求标识、节点信息和阶段结果
- 热更新是否有成功、失败、回退三类观测信号
- 缓存和配置切换是否声明最大生效时延
- 出现短暂新旧并存时，行为是否有明确定义
