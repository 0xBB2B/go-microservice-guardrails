# 配置与共享包边界

## 主题目标

把服务私有配置留在服务内部，把共享能力限制为无业务语义的基础设施，避免为了去重破坏边界。

## 必须遵守

- 每个服务的配置都放在 `services/<service>/internal/config`
- 每个服务只保留自己的配置模型、加载逻辑和必填校验
- `pkg/` 只承载跨服务复用且无业务语义的基础设施能力
- 只有稳定复用、且不依赖服务专属配置模型、业务规则或内部数据结构的代码，才可以进入 `pkg/`
- 边界优先于去重；即使代码相似，也不代表应该抽到共享包

## 常见错误

- 在一个服务的 `config` 包里声明另一个服务的配置类型
- 把服务专属鉴权流程、领域模型或业务规则抽到 `pkg/`
- 因为“看起来很像”就合并两份服务私有实现
- 用共享包承载服务私有配置模型

## 落地检查点

- 新增配置字段时，是否只改当前服务的 `internal/config`
- 抽共享包前，是否能证明它不携带业务语义
- 共享包 import 是否依赖某个服务的内部类型或私有配置
- 服务启动流程是否从本服务 `internal/config` 读取配置，再从 `pkg/` 引入基础设施
