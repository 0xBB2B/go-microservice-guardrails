# 数据库、部署与测试

## 主题目标

让数据权限、部署资源和测试分层都围绕服务边界展开，避免运行时越权和联调资源散落。

## 必须遵守

- 一服务一数据边界，关系型场景至少做到一服务一 schema 或等价逻辑边界
- 一服务一组运行时凭据，运行时只注入本服务真正需要的 DSN
- 跨服务管理动作走接口，不直连其他服务数据库改数据
- MySQL 统一使用 `utf8mb4` 与 `utf8mb4_0900_ai_ci`
- 联调与部署辅助资源统一收敛到 `ops/`
- Docker 构建和多模块协同以仓库根和 `go.work` 为基础
- 测试按服务内、共享层、跨服务层拆分，跨服务测试统一放在 `ops/tests/`

## 常见错误

- 给某个服务注入其他服务私有库的运行时 DSN
- 正常请求路径持有高权限管理账号
- 服务目录和 `ops/` 同时维护重复联调入口
- 把跨服务回归测试散落到单个服务目录
- 依赖隐式字符集或排序规则，不显式声明数据库约束

## 落地检查点

- 新增数据库访问时，是否只使用本服务的数据边界和最小权限账号
- 新增联调资源时，是否统一进入 `ops/`
- 新增测试时，是否落在正确层级
- 新增镜像或构建脚本时，是否正确利用仓库根与 `go.work`
- DSN、schema、字符集与排序规则是否显式声明
