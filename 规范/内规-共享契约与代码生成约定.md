# 共享契约与代码生成约定

## 1. 目标与适用范围

本文档定义当前仓库的共享契约组织方式、命名规则、代码生成规则和引用规则，作为本仓库的现行契约内规。

为避免契约规范绑定某个具体项目名或具体服务名，本文统一使用 `<workspace-name>`、`<service>`、`<public-service>`、`<admin-service>`、`<public-method>`、`<admin-method>`、`<business-request>`、`<business-field>`、`<route-path>` 等代号描述。

本文档适用于当前仓库内所有需要被多个模块共同消费的协议定义，包括但不限于：

- 服务间 gRPC 契约
- 由 proto 驱动的 HTTP 映射
- 内部管理接口契约

本文档不是只为某几个固定服务编写。后续新增任意服务时，也应遵守同一套契约治理方式。

## 2. 契约真源约定

`contracts/proto/` 是当前仓库共享契约唯一来源。

这意味着：

1. 共享 proto 只能在 `contracts/proto/` 下维护真源。
2. 服务代码只消费 `contracts/gen/go/` 中的生成结果。
3. 不允许在服务目录内长期维护另一份共享契约真源。

## 3. 命名空间与目录规则

### 3.1 服务命名空间

共享契约统一按“服务拥有接口”的原则组织目录：

```text
contracts/proto/<service>/v1/
```

例如：

- `contracts/proto/<service>/v1/`

### 3.2 proto package 规则

proto package 统一使用：

```text
<service>.v1
```

例如：

- `<service>.v1`

不再使用与目录所有权不一致的 package 命名。

### 3.3 文件命名规则

proto 文件名应表达接口职责，而不是表达调用关系。

推荐规则如下：

1. 服务对外认证或公钥类契约：`<service>_auth.proto`
2. 服务主业务接口契约：`<service>_service.proto`
3. 服务内部管理接口契约：`<service>_admin_service.proto`

当前现行契约命名示例如下：

- `contracts/proto/<service>/v1/<service>_auth.proto`
- `contracts/proto/<service>/v1/<service>_service.proto`
- `contracts/proto/<service>/v1/<service>_admin_service.proto`

## 4. Go 生成路径规则

共享契约生成代码统一落在：

```text
contracts/gen/go/<service>/v1/
```

`go_package` 统一指向：

```text
<workspace-name>/contracts/gen/go/<service>/v1;<service>v1
```

实际仓库中的 module path 应按上面的 `<workspace-name>` 形式维护，而不是写死某个具体项目名。

## 5. 当前现行契约基线

### 5.1 认证或公钥契约基线

某个服务如果需要向其他服务分发认证或公钥相关能力，其契约应满足：

- 文件名按 `<service>_auth.proto` 组织
- package 按 `<service>.v1` 组织
- 服务名应按 `<service>` 派生，并清楚表达“认证”或“公钥分发”职责

其职责是暴露当前服务可共享给其他服务消费的认证相关接口，例如公钥获取能力。

### 5.2 业务与管理契约基线

某个服务如果同时暴露公开业务接口和内部管理接口，建议至少拆成两类契约：

1. 主业务契约：`<service>_service.proto`
2. 内部管理契约：`<service>_admin_service.proto`

这两类契约都属于 `<service>` 自己拥有的接口面，不应由调用方服务定义。

### 5.3 HTTP 路由基线

当前仓库中，由 proto 驱动的公开 HTTP 映射应以契约定义为准。

现行基线如下：

1. `<public-method>` 通过 `google.api.http` 暴露 `<route-path>`
2. 公开路径和字段映射以 proto 为唯一准源

## 6. 请求与字段定义基线

### 6.1 业务 request 只保留业务字段

业务 request 应只承载业务参数，不承载入口层上下文。

当前现行基线如下：

- `<service>.v1.<business-request>` 只保留业务字段，例如 `<business-field>`

### 6.2 用户上下文通过 metadata 透传

当前仓库中，入口层注入的用户上下文字段不进入 `<business-request>`，而是通过 metadata 透传。

当前现行字段基线如下：

- `x-service-token`
- `x-user-id`
- `x-user-authenticated`

如果未来新增更多上下文字段，也必须优先明确它们属于 metadata 还是业务 payload，不能混用。

### 6.3 内部管理契约独立定义

内部管理动作应有单独契约，不与公开业务接口混在同一方法中。

当前示例：

- `<admin-method>` 属于内部管理契约

未来其他服务如果存在管理动作，也应优先新增显式管理契约，而不是扩展公开业务 request 承担管理语义。

## 7. 代码生成规则

### 7.1 生成入口

Buf 是当前仓库共享契约的统一生成入口。

修改 proto 后，必须同步执行生成与校验。

### 7.2 生成物提交规则

修改 `contracts/proto/` 后，必须同步更新：

1. `contracts/gen/go/`
2. 相关契约测试
3. 相关说明文档

不允许只改 proto 而不提交生成物。

### 7.3 契约变更联动规则

契约变更至少应联动以下内容：

1. 服务端实现引用
2. 调用方引用
3. 契约测试
4. README 或现行规范文档

## 8. 引用规则

当前仓库中，服务代码只允许引用：

```text
<workspace-name>/contracts/gen/go/...
```

不允许：

1. 在服务代码中直接维护另一份共享 proto。
2. 让调用方自行复制并长期维护共享契约真源。
3. 让代码与文档对同一接口给出不同字段定义或不同路径定义。

## 9. 契约测试规则

契约测试至少应锁定以下关键内容：

1. 关键 request 字段数量和字段名。
2. 关键 HTTP 路由映射。
3. 关键方法和消息类型是否存在。
4. 关键管理接口是否可被正确引用。

## 10. 禁止事项

必须禁止以下做法：

1. 使用与目录所有权不一致的 proto package。
2. 让调用方反向决定被调用方业务字段语义。
3. 在不同文档中为同一接口写出不同路径。
4. 在代码和文档中为同一 request 写出不同字段集合。
5. 修改 proto 后不更新生成物和测试。

## 11. 与其他规范的关系

- 服务边界和接口所有权，可配合《通用-服务边界与接口所有权》使用。
- 服务间认证、授权和上下文透传，可配合《服务间通信与权限验证约定》使用。
- 当前仓库目录与模块角色，可配合《内规-仓库模块与目录约定》使用。

## 12. 一句话结论

**共享契约只在 `contracts/proto/` 维护真源，谁拥有接口谁拥有契约，代码只引用 `contracts/gen/go/`，路径、字段和生成物必须始终与契约定义保持一致。**
