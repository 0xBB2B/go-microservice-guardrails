# 仓库模块与目录约定

## 1. 目标与适用范围

本文档定义当前仓库的模块划分、目录职责和工作区边界，作为本仓库的现行目录规范。

本文档只约束当前仓库，不作为跨项目通用模板。

为避免目录规范绑定某个具体服务名，本文统一使用 `<service>`、`<service-a>`、`<service-b>` 代表任意服务目录名。

本文档重点回答以下问题：

1. 当前仓库有哪些模块。
2. 每个模块和目录各自负责什么。
3. 新增代码、新增服务和新增联调资源时应放在哪里。
4. 哪些文档属于现行规范，哪些文档只是历史记录。

## 2. 仓库模块分层

当前仓库采用 `go.work` 管理多模块工作区。模块划分不是为任何具体服务名定制，而是为后续持续新增服务预留统一骨架。

现行模块如下：

| 模块 | 角色 |
| --- | --- |
| `contracts/` | 共享契约模块，维护 proto、Buf 配置和生成代码 |
| `pkg/` | 共享基础设施模块，维护跨服务复用的底层能力 |
| `services/` | 独立服务模块根目录，子目录统一使用 `services/<service>/` |
| `ops/tests/` | 跨服务部署校验与 E2E 测试 |

根目录不再承载单一业务 Go module。

未来如果新增任意服务，也应按相同分层进入：

- 新服务代码进入 `services/<service>/`
- 共享契约进入 `contracts/`
- 共享基础设施能力进入 `pkg/`
- 多服务联调资源进入 `ops/`

## 3. 根目录职责

根目录当前承担以下职责：

1. 承载 `go.work`。
2. 提供总 README。
3. 存放现行规范与运行手册。
4. 作为多服务联调与镜像构建的公共上下文根。

根目录不应承担以下职责：

1. 作为某个业务服务的源码目录。
2. 存放服务私有实现代码。
3. 存放共享 proto 以外的跨服务业务源码。

## 4. 标准目录职责

### 4.1 `contracts/`

`contracts/` 是当前仓库共享契约唯一来源，负责：

1. 维护共享 proto。
2. 维护 Buf 配置。
3. 维护统一生成的 Go 协议代码。

不允许在服务目录内维护另一份共享 proto 真源。

### 4.2 `pkg/`

`pkg/` 是共享基础设施模块，负责：

1. 日志。
2. MySQL 连接与等待。
3. OTel 初始化。
4. 其他无业务语义的跨服务基础能力。

`pkg/` 不负责：

1. 领域模型。
2. 服务私有配置模型。
3. 服务私有鉴权流程。
4. 服务专属仓储与业务规则。

### 4.3 `services/`

`services/` 下每个 `services/<service>/` 子目录都是一个独立服务模块。

后续可以持续扩展多个独立服务目录，但都必须遵守同一套目录骨架和边界规则。

无论服务名称是什么，每个服务目录都应承担：

1. 自己的入口代码。
2. 自己的私有实现。
3. 自己的配置模型。
4. 自己的 README、`go.mod`、`Makefile`、`Dockerfile`。

### 4.4 `ops/`

`ops/` 统一承载多服务联调与运维辅助资源，负责：

1. `docker-compose.yml`
2. MySQL 初始化脚本
3. OTel Collector 配置
4. 跨服务测试入口

`ops/` 不负责：

1. 服务私有业务实现
2. 服务私有单元测试
3. 共享契约定义

### 4.5 `docs/`

`docs/` 根目录只放当前仓库的现行文档，例如：

1. 通用模板
2. 仓库内规
3. 运行手册

## 5. 服务目录约定

每个服务目录至少应具备以下结构：

1. `cmd/<service>/`：服务入口
2. `internal/`：服务私有实现
3. `README.md`
4. `go.mod`
5. `Makefile`
6. `Dockerfile`
7. `.env.example`

当前服务内实现约定如下：

- 配置目录统一使用 `internal/config`
- 服务私有逻辑统一留在 `internal/`

## 6. 工作区约定

### 6.1 `go.work` 角色

根目录 `go.work` 是当前多模块工作区的唯一工作区定义。

其职责是：

1. 把 `contracts`、`pkg`、各服务模块和 `ops/tests` 聚合到同一工作区。
2. 提供本地联调开发体验。
3. 避免各模块在日常开发中维护额外 `replace` 配置。

### 6.2 新模块加入规则

新增可独立构建的 Go 子模块时，应显式加入 `go.work`。

如果新增内容不需要独立模块边界，则不应为了“方便”额外创建 module。

## 7. 允许的依赖关系

当前仓库允许以下依赖关系：

1. 服务依赖 `contracts` 生成包。
2. 服务依赖 `pkg` 中的共享基础设施包。
3. `ops/tests` 依赖运行中的服务和公开契约。
4. 各服务通过网络协议互相调用。

## 8. 禁止的依赖关系

当前仓库必须禁止以下关系：

1. 任意 `services/<service-a>` 直接 import 另一个 `services/<service-b>` 的内部实现。
2. 任意服务把另一个服务当作本地库直接调用，而不是通过公开契约协作。
3. 在 `pkg/` 中放入领域逻辑、服务私有配置或服务私有鉴权流程。
4. 在 `ops/` 中放入服务私有业务代码。
5. 在服务目录中维护与 `contracts/` 冲突的共享契约真源。

## 9. 新增服务落地约定

未来新增服务时，应遵守以下规则：

1. 新服务放在 `services/<service>/`
2. 新服务拥有自己的 `go.mod`
3. 新服务拥有自己的 `cmd/<service>/` 和 `internal/`
4. 服务间共享契约优先进入 `contracts/`
5. 服务间共享基础设施能力优先进入 `pkg/`
6. 多服务联调改动统一落在 `ops/`

新增服务时，不应复制某个既有服务的业务职责命名。应该复制的是目录骨架和边界规则，而不是具体业务语义。

## 10. 文档治理约定

### 10.1 现行文档

以下内容属于现行规范与现行手册，需随代码演进维护：

1. `docs/` 根目录文档
2. 根目录 `README.md`
3. 服务目录下的 `README.md`
4. `contracts/README.md`
5. `pkg/README.md`

### 10.2 历史记录

以下内容属于历史设计与实施记录，不作为现行规范解释来源：

1. `docs/superpowers/specs/`
2. `docs/superpowers/plans/`

如历史记录与当前代码不一致，优先以当前代码、测试和现行文档为准。

## 11. 落地检查清单

新增目录或模块时，建议至少检查：

1. 它是否真的需要独立模块边界。
2. 它是否被放在了正确层级。
3. 它是否引入了跨服务源码依赖。
4. 它是否让 `contracts`、`pkg`、`services`、`ops` 的边界变得模糊。

## 12. 一句话结论

**当前仓库按 `contracts`、`pkg`、`services`、`ops` 四层划分职责；服务名称可以持续扩展，但分层规则不变：共享能力进共享层，服务私有实现留在服务层，联调资源统一收敛到 `ops`。**
