# 服务边界与接口所有权通用模板

## 1. 目标与适用范围

本文档定义一套可跨项目复用的服务边界与接口所有权模板，适用于所有存在“一个服务调用另一个服务”关系的系统。

本文档重点约束三类问题：

1. 服务边界应如何划分。
2. 接口定义与字段语义应归谁拥有。
3. 服务之间允许和禁止建立哪些依赖关系。

本文档不规定具体语言、框架、仓库目录，也不替代认证、授权、数据库或部署规范。它只回答一个基础问题：

**服务应该如何彼此协作，而不因为实现细节复用而破坏边界。**

## 2. 角色定义

为避免文档绑定具体项目，统一使用以下抽象角色：

| 角色 | 定义 |
| --- | --- |
| `入口服务` | 接收外部流量，负责统一入口能力的服务 |
| `领域服务` | 拥有特定业务能力、业务数据和业务语义的服务 |
| `调用方服务` | 主动向其他服务发起调用的服务 |
| `被调用方服务` | 暴露内部接口，并定义接口字段语义和处理规则的服务 |
| `共享基础设施模块` | 提供日志、追踪、数据库连接、通用中间件等无业务语义能力的模块 |

说明：

- `入口服务` 通常也是 `调用方服务`。
- `领域服务` 在不同链路中既可能是 `被调用方服务`，也可能继续充当 `调用方服务`。
- `共享基础设施模块` 不是业务服务，不拥有业务接口语义。

## 3. 服务边界基本原则

### 3.1 自治原则

每个服务都必须对以下内容负责：

1. 自己的业务逻辑。
2. 自己的数据模型。
3. 自己的配置模型。
4. 自己的运行行为与错误语义。

不允许将这些职责依赖为“另一个服务的内部实现”。

### 3.2 契约协作原则

服务之间必须通过公开契约协作，而不是通过直接引用对方内部代码协作。

公开契约可以是：

- gRPC / HTTP / GraphQL / 消息协议定义
- 明确发布的 SDK 或客户端契约
- 经治理的事件结构

不属于公开契约的内容包括：

- 对方服务的内部函数
- 对方服务的内部数据结构
- 对方服务的仓储实现
- 对方服务的私有配置

### 3.3 边界优先原则

当“代码复用”与“服务边界清晰”发生冲突时，优先保留服务边界。

原因很简单：

- 边界模糊会放大耦合成本。
- 一次“为了方便”的跨边界复用，往往会固化成长期依赖。
- 服务一旦互相把对方当作本地库，独立演进、独立测试和独立发布都会失真。

### 3.4 共享能力收敛原则

只有同时满足以下条件的能力，才适合进入共享基础设施模块：

1. 不携带领域语义。
2. 可以被多个服务稳定复用。
3. 不会反向要求服务暴露自己的内部模型。

常见适合共享的能力：

- 日志
- 链路追踪
- 数据库连接
- 重试与超时控制
- 通用中间件

常见不适合共享的能力：

- 业务规则
- 领域模型
- 服务私有配置模型
- 服务私有鉴权流程
- 服务专属仓储实现

## 4. 接口所有权原则

### 4.1 定义所有权

谁提供接口，谁拥有接口定义。

这意味着：

1. 被调用方定义方法名、字段名和字段语义。
2. 调用方只能消费接口，不能反向决定被调用方字段应该表达什么。
3. 字段的解释权始终属于接口提供方，而不是入口层或聚合层。

### 4.2 语义所有权

接口字段不只是传输结构，也是业务语义的载体。

因此必须坚持：

- 业务参数的含义由被调用方定义。
- 业务返回结果的结构由被调用方定义。
- 调用方不能长期持有“代替下游解释字段语义”的能力。

### 4.3 兼容性所有权

谁拥有接口，谁负责接口演进与兼容策略，包括：

1. 新字段如何增加。
2. 旧字段何时废弃。
3. 破坏性变更如何迁移。
4. 多版本如何共存。

调用方不能把自己的兼容成本转嫁给被调用方内部实现。

## 5. 标准职责划分模板

### 5.1 入口服务职责

入口服务通常负责：

1. 接收外部流量。
2. 做统一认证与基础校验。
3. 识别用户上下文或租户上下文。
4. 做协议适配、请求转发和统一可观测性接入。

入口服务通常不应负责：

1. 定义下游业务字段语义。
2. 长期维护下游领域参数映射规则。
3. 直接实现下游领域服务的数据查询与业务判断。

### 5.2 领域服务职责

领域服务通常负责：

1. 拥有业务接口定义。
2. 定义请求参数和响应结构的业务语义。
3. 执行业务校验与业务逻辑。
4. 管理领域数据的读写与返回。

### 5.3 共享基础设施模块职责

共享基础设施模块通常负责：

1. 提供无业务语义的底层能力。
2. 为多个服务提供稳定复用点。
3. 降低重复实现，但不改变服务边界归属。

共享基础设施模块不应负责：

1. 组合多个服务的业务流程。
2. 持有某个服务的专属配置模型。
3. 持有某个服务专属的领域规则。

## 6. 允许的依赖关系

推荐允许以下依赖关系：

1. 服务依赖共享契约。
2. 服务依赖共享基础设施模块。
3. 服务依赖外部基础设施能力。
4. 服务通过网络协议调用其他服务。

这些依赖成立的前提是：

- 依赖边界明确。
- 所有权明确。
- 调用关系通过公开契约表达。

## 7. 禁止的依赖关系

必须禁止以下关系：

1. 服务源码级直接依赖其他服务内部实现。
2. 将其他服务作为本地库直接调用。
3. 直接读写其他服务的私有数据存储。
4. 将服务私有配置、领域模型、仓储实现放入共享基础设施模块。
5. 由调用方反向决定被调用方的业务参数和返回语义。

## 8. 接口演进规则

### 8.1 向后兼容优先

接口演进应优先选择向后兼容方式：

1. 新增字段优先使用追加方式。
2. 旧字段废弃前应保留兼容窗口。
3. 默认值和行为变化必须明确记录。

### 8.2 破坏性变更显式化

以下变更应视为破坏性变更：

- 删除字段
- 更改字段语义
- 更改错误语义
- 更改必填规则
- 更改路径或方法绑定关系

破坏性变更必须显式治理，不能依赖调用方“自行适配”。

### 8.3 文档与测试同步

接口一旦变化，应同步更新：

1. 契约定义。
2. 契约测试。
3. 说明文档。
4. 调用方接入验证。

## 9. 例外处理规则

在真实项目中，可能存在短期跨边界例外，但必须满足以下条件：

1. 例外原因明确。
2. 影响范围明确。
3. 失效条件明确。
4. 回收计划明确。

禁止把“临时方便”固化为长期默认模式。

## 10. 落地检查清单

新增服务时，建议至少检查：

1. 该服务是否拥有清晰的业务边界。
2. 该服务是否只暴露自己的公开契约。
3. 该服务是否避免依赖其他服务内部实现。
4. 该服务的配置、数据和运行行为是否由自己负责。

新增接口时，建议至少检查：

1. 接口所有权是否归被调用方。
2. 字段语义是否由提供方定义。
3. 调用方是否只消费公开契约。
4. 契约、测试和文档是否同步更新。

抽取共享代码时，建议至少检查：

1. 该能力是否真的无业务语义。
2. 该能力是否已被多个服务稳定复用。
3. 抽取后是否会迫使服务暴露内部模型。

## 11. 与其他规范的关系

- 服务间认证、方法授权和用户上下文透传，可配合《服务间通信与权限验证约定》使用。
- 数据边界与最小权限控制，可配合《数据访问最小权限通用模板》使用。
- 可观测性、缓存刷新和热更新策略，可配合《可观测性与热更新通用模板》使用。

## 12. 一句话结论

**服务之间只通过公开契约协作，不通过内部实现协作；谁提供接口，谁拥有接口；谁拥有业务，谁拥有语义。**
