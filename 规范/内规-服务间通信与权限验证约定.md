# 服务间通信与权限验证约定

## 1. 目标与适用范围

本文档定义当前仓库的服务间通信、服务认证、方法授权和用户上下文透传规则，作为本仓库的现行内规。

为避免规范绑定某个具体示例项目、具体服务名或具体业务方法，本文统一使用通用代号描述。实际落地时，应由各服务在自己的 README、配置说明和契约文档中建立代号到真实名称的映射。

本文重点回答四个问题：

1. 服务之间如何证明彼此身份。
2. 谁拥有方法级授权的决定权。
3. 哪些用户字段允许透传，以及透传到哪里。
4. 密钥轮换和缓存失败时按什么规则处理。

## 2. 代号约定

本文统一使用以下占位代号：

| 代号 | 含义 |
| --- | --- |
| `<workspace-name>` | 当前仓库或工作区名称 |
| `<entry-service>` | 接收外部流量、识别用户上下文并向下游发起调用的入口服务 |
| `<caller-service>` | 当前这一次内部调用中的调用方服务 |
| `<callee-service>` | 当前这一次内部调用中的被调用方服务 |
| `<key-provider-service>` | 对外分发当前公钥的服务 |
| `<public-service>` | 被调用方暴露的公开业务服务 |
| `<admin-service>` | 被调用方暴露的内部管理服务 |
| `<public-method>` | 公开业务方法名 |
| `<admin-method>` | 内部管理方法名 |
| `<auth-service>` | 用于公钥分发的认证支撑服务 |
| `<get-public-key-method>` | 公钥分发方法名 |

本文出现的 `<caller-service>`、`<callee-service>`、`<key-provider-service>` 只是角色代号，不代表固定是三个不同服务。在具体链路中，它们可以映射到同一个真实服务，也可以映射到不同真实服务。

## 3. 标准调用链表达方式

当前仓库中的链路应统一按如下抽象形式表达：

- 外部请求 -> `<entry-service>` -> `<callee-service>.<public-service>/<public-method>`
- 内部管理请求 -> `<entry-service>` -> `<callee-service>.<admin-service>/<admin-method>`
- `<callee-service>` -> `<key-provider-service>.<auth-service>/<get-public-key-method>`

这样写的目的不是抽象掉真实实现，而是要求所有规范、说明文档和设计文档都优先描述角色职责，而不是把规则绑死在某个具体服务名上。

## 4. 通信基线

### 4.1 业务参数与上下文字段分层

当前仓库明确区分“业务参数”和“链路上下文”：

- 业务参数进入 proto request。
- 服务身份和用户上下文进入 gRPC metadata。

统一表达方式如下：

- `<callee-service>.<public-service>/<public-method>` 的 request 只承载业务字段。
- `<callee-service>.<admin-service>/<admin-method>` 的 request 只承载管理动作字段。
- `x-service-token`、`x-user-id`、`x-user-authenticated` 不进入业务 request。

### 4.2 当前统一字段

当前仓库统一以下 metadata 字段名：

| 字段 | 作用 |
| --- | --- |
| `x-service-token` | 服务身份令牌 |
| `x-user-id` | 用户 ID |
| `x-user-authenticated` | 登录态结果 |

除上述字段外，`x-user-scopes`、`x-tenant-id`、`traceparent` 等字段不属于当前强制基线。后续如果要新增，必须同步更新代码、测试和本文档，不能只改其中一处。

### 4.3 固定处理顺序

当前仓库所有受保护的业务调用都必须遵守以下顺序：

1. `<caller-service>` 读取自己当前生效的私钥。
2. `<caller-service>` 签发短时效 `x-service-token`。
3. 只有在请求确实源于用户时，`<caller-service>` 才附带最小用户上下文。
4. `<caller-service>` 通过统一 client、入口适配层或 interceptor 注入 metadata，不允许业务代码自行拼装。
5. `<callee-service>` 先完成服务认证。
6. `<callee-service>` 再完成方法授权。
7. 只有认证和授权都通过后，`<callee-service>` 才把可信调用方身份和用户上下文注入内部 `context`。
8. 最后才进入业务 handler。

`<callee-service> -> <key-provider-service>.<auth-service>/<get-public-key-method>` 属于公钥分发支撑链路，当前不复用上述服务令牌认证流程。它的职责只是在验签前提供当前公钥，不承载业务参数和用户上下文。

## 5. 服务认证规则

### 5.1 令牌基线

当前仓库使用 `JWT service token` 表达服务身份，现行基线如下：

| 项目 | 当前基线 |
| --- | --- |
| 签名算法 | `Ed25519`，JWT `EdDSA` |
| 令牌位置 | gRPC metadata：`x-service-token` |
| 调用方标识 | 自定义 claim：`svc` |
| 时间字段 | `iat`、`exp` |
| 默认有效期 | 30 秒 |
| 签发主体 | 当前链路中的 `<caller-service>` |

当前实现尚未把 `aud`、`nbf`、`jti`、`kid` 作为强制字段。如果后续要引入这些字段，必须同步更新：

1. 调用方签发逻辑。
2. 被调用方验签逻辑。
3. 契约测试与联调测试。
4. 本文档和相关说明文档。

### 5.2 公钥分发与缓存基线

当前仓库的公钥分发方式固定如下：

1. `<caller-service>` 的私钥和公钥都保存在 `<caller-service>.service_keys`。
2. `<key-provider-service>` 通过 `<auth-service>/<get-public-key-method>` 对外暴露当前生效公钥。
3. `<callee-service>` 通过 gRPC 客户端按需拉取 `<caller-service>` 的公钥。
4. `<callee-service>` 以服务名为键把公钥缓存在本地内存。
5. 验签失败时允许刷新一次公钥，并且只重试一次。

`<auth-service>/<get-public-key-method>` 是支撑接口，不适用 `<callee-service>.call_policies` 的方法授权模型，也不透传用户上下文。

### 5.3 入站验签顺序

`<callee-service>` 必须按以下顺序处理入站 gRPC 请求：

1. 从 metadata 提取 `x-service-token`。
2. 从未验签 token 中解析 `svc`，仅用于定位公钥。
3. 根据 `svc` 从本地缓存读取公钥；缓存未命中时回源拉取。
4. 使用 `EdDSA` 和当前公钥验签，并校验 token 有效性。
5. 如果验签失败，则刷新一次公钥并仅重试一次。
6. 从 gRPC 全方法名提取短方法名。
7. 按 `<callee-service>` 自己的授权策略判断该方法是否允许当前服务调用。
8. 只有上述步骤全部通过后，才把 `caller_service` 和用户上下文写入内部 `context`。

必须明确：

- `svc` 可以在验签前被解析出来用于查找公钥，但在验签成功前，它不是可信身份，不能被业务逻辑直接使用。

### 5.4 禁止事项

必须禁止以下做法：

1. 在业务 handler 中直接读取 `x-service-token` 自行做鉴权判断。
2. 在完成服务认证前直接信任 `x-user-id` 或 `x-user-authenticated`。
3. 由业务代码自行拼装 `x-service-token`。
4. 未来新增服务时复用别的服务的私钥或公钥表。

## 6. 方法授权规则

### 6.1 授权归属

当前仓库中，被调用方始终拥有对自身 gRPC 接口的授权决定权。

这意味着：

1. 是否允许调用 `<public-method>`，由 `<callee-service>` 决定。
2. 是否允许调用 `<admin-method>`，仍然由 `<callee-service>` 决定。
3. `<caller-service>` 可以发起管理调用，但不能自我声明“我有权限调用”。

### 6.2 策略真源

当前仓库的授权真源统一表达为 `<callee-service>.call_policies`，现行字段如下：

| 字段 | 含义 |
| --- | --- |
| `caller_service` | 调用方服务名 |
| `allowed_method` | 允许调用的方法短名 |
| `status` | `enabled` / `disabled` |
| `updated_at` | 最近更新时间 |

其中 `allowed_method` 必须使用与拦截器一致的方法短名，不允许把完整 gRPC 方法路径和短方法名混用。

### 6.3 判断规则

`<callee-service>` 必须遵守以下授权规则：

1. 默认拒绝。
2. 只有命中 `status = 'enabled'` 的显式规则时才放行。
3. `caller_service` 和 `allowed_method` 必须同时匹配。
4. `<admin-method>` 的状态更新只能在受控枚举范围内变化，例如 `enabled` 或 `disabled`。

### 6.4 缓存规则

`<callee-service>` 当前使用本地内存缓存授权策略，现行规则如下：

| 项目 | 当前基线 |
| --- | --- |
| 缓存位置 | `<callee-service>` 进程内存 |
| 刷新方式 | 定时轮询数据库 |
| 刷新周期 | 60 秒 |
| 启动要求 | 首次加载失败则服务启动失败 |
| 轮询失败行为 | 保留上一次成功快照，等待下一轮刷新 |

未来如果新增其他被调用方服务，也必须维护自己的授权真源，不能把所有方法授权集中堆到某个入口服务。

## 7. 用户上下文透传规则

### 7.1 最小用户上下文

当前仓库只认可以下两个用户字段：

| 字段 | 说明 |
| --- | --- |
| `x-user-id` | 已识别用户 ID |
| `x-user-authenticated` | 是否已登录 |

这两个字段都不是前端原始访问令牌，而是 `<entry-service>` 识别外部请求后得到的用户结果。

### 7.2 透传范围

当前仓库中，用户上下文只用于 `<entry-service> -> <callee-service>` 的公开业务链路。

现行规则如下：

1. `<entry-service>` 从外部请求中识别用户态。
2. `<entry-service>` 把识别结果写入 `x-user-id` 与 `x-user-authenticated`。
3. `<callee-service>` 在服务认证通过后读取这两个字段，并写入内部 typed context。
4. `<admin-service>/<admin-method>` 默认不依赖用户上下文，除非有额外显式约束。
5. 业务 proto request 不允许新增 `user_id`、`authenticated` 这类入口层字段。

### 7.3 最小化原则

必须遵守以下原则：

1. 只透传验证后的用户结果，不透传前端原始 token。
2. 只透传当前下游确实需要的最小字段。
3. 缺少用户字段时，可以按匿名用户处理，但不能绕过服务认证。
4. 业务代码应从内部 context 读取可信用户信息，而不是重新读取原始 metadata。

### 7.4 多跳扩展规则

未来如果出现 `<entry-service> -> <service-a> -> <service-b>` 这类多跳链路，必须继续遵守：

1. 每一跳都重新签发自己的 `x-service-token`。
2. 每一跳都重新判断哪些用户字段值得继续传播。
3. 下游服务只信任直接调用自己的那个服务。
4. 不允许把上游服务签发的 `x-service-token` 原样转发给下游。

## 8. 密钥轮换与容错规则

### 8.1 轮换模型

当前仓库中，服务密钥轮换按如下通用模型处理：

1. 为 `<caller-service>` 生成一对新的 `Ed25519` 公私钥。
2. 以 `pending` 状态写入 `<caller-service>.service_keys`。
3. 把旧的 `active` 密钥标记为 `revoked`。
4. 把新的 `pending` 密钥提升为 `active`。
5. `<caller-service>` 通过定时轮询重新加载当前 `active` 私钥。
6. `<callee-service>` 在缓存未命中或验签失败时拉取新的 `active` 公钥。

### 8.2 当前实现约束

当前实现还存在以下明确约束，不能被文档美化掉：

1. 私钥刷新周期当前为 60 秒。
2. 公钥缓存当前没有独立 TTL，而是依赖缓存命中与验签失败时的按需刷新。
3. 当前实现没有长期双活公钥窗口。
4. 当前 token TTL 只有 30 秒，轮换策略依赖短时效 token 才能稳定工作。

因此：

- 不允许在不调整轮换策略的前提下随意拉长 token TTL。
- 如果未来需要更长 TTL 或更频繁的密钥切换，必须先补齐兼容窗口设计，再修改实现和文档。

### 8.3 失败语义

当前仓库的主要失败语义如下：

| 场景 | 当前结果 |
| --- | --- |
| `<caller-service>` 无法读取当前私钥 | 出站失败 |
| 缺少 `x-service-token` | `gRPC Unauthenticated` |
| token 非法或验签失败且刷新后仍失败 | `gRPC Unauthenticated` |
| 公钥加载或刷新失败 | 当前实现返回 `gRPC Unauthenticated` |
| 方法未授权 | `gRPC PermissionDenied` |
| 授权策略首次加载失败 | `<callee-service>` 启动失败 |

这些语义反映的是当前代码现状。后续如果要改成更细的错误分类，必须同步修改实现、测试和文档。

## 9. 工程约束

当前仓库必须固定以下工程边界：

1. 出站鉴权注入统一收敛在请求增强器、共享 client 或 interceptor，不能散落到业务代码。
2. 入站服务认证统一收敛在 middleware 或 interceptor，不能下沉到 handler。
3. 业务 handler 只读取已经注入到 context 的可信身份，不直接读取原始 metadata。
4. 服务身份字段、用户上下文字段、方法授权规则一旦变化，必须同步更新代码、测试和文档。
5. 当前联调环境中使用的传输安全配置只代表 demo 现状，不自动等同于生产安全基线。

## 10. 新增服务时的落地规则

未来新增服务时，至少必须满足以下要求：

1. 新调用方服务拥有自己的私钥和公钥分发能力。
2. 新被调用方服务拥有自己的授权真源和授权缓存。
3. 新链路继续沿用 `x-service-token`、最小用户上下文和逐跳授权规则。
4. 新增 claim、新增 header 或新增多跳传播字段时，必须同步更新契约、代码、测试、运维配置和本文档。

## 12. 一句话结论

**当前仓库的服务间通信内规以“调用方签发短时效 `Ed25519` 服务令牌、被调用方独立验签并按本地策略表授权”为基线；业务参数进 proto，服务身份与用户上下文进 metadata，未来新增服务也必须逐跳签发、逐跳授权、逐跳最小透传。**
