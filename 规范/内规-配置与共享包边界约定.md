# 配置与共享包边界约定

## 1. 目标与适用范围

本文档定义当前仓库中服务私有配置与共享基础设施包的边界规则，作为本仓库的现行内规。

本文档适用于当前和未来所有 `services/<service>/` 模块。

本文档重点解决两个问题：

1. 什么内容必须留在服务内部。
2. 什么内容可以沉淀到 `pkg/` 作为共享基础设施能力。

## 2. 基本原则

### 2.1 配置属于服务私有实现

每个服务的启动配置、环境变量模型和必填校验都属于服务私有实现。

因此：

1. 服务配置必须放在 `services/<service>/internal/config`
2. 每个服务只保留自己的配置模型和加载逻辑
3. 不允许在本服务 `config` 包中声明别的服务的配置类型

### 2.2 共享包只承载无业务语义能力

`pkg/` 只承载跨服务复用、且不带业务语义的基础设施能力。

只有满足以下条件的代码，才适合进入 `pkg/`：

1. 被多个服务稳定复用
2. 不依赖某个服务专属配置模型
3. 不依赖某个服务的业务规则
4. 不依赖某个服务的内部数据结构

### 2.3 边界优先于去重

即使两份代码暂时看起来很像，也不代表应该抽到共享包。

如果一段代码承载的是服务专属语义，即使存在重复，也应优先保留在服务内部。

## 3. 服务配置目录约定

每个服务统一使用：

```text
services/<service>/internal/config
```

该目录负责：

1. 定义本服务配置结构体
2. 读取本服务环境变量
3. 做本服务必填项校验
4. 为本服务入口提供统一配置装载入口

不负责：

1. 定义其他服务的配置模型
2. 提供跨服务通用基础设施能力
3. 承载业务逻辑

## 4. 共享包目录约定

当前 `pkg/` 下现行共享包包括：

- `pkg/logging`
- `pkg/mysql`
- `pkg/otel`

后续如果新增共享包，也应遵守同样边界：先证明它是无业务语义的基础设施能力，再进入 `pkg/`。

## 5. 什么可以进入 `pkg/`

以下内容适合进入 `pkg/`：

1. 结构化日志封装
2. Trace 关联日志辅助函数
3. MySQL 连接与等待逻辑
4. OTel 初始化逻辑
5. 通用中间件或基础工具

这些能力的共同特征是：

- 不拥有某个业务接口语义
- 不拥有某个服务专属配置模型
- 不拥有某个服务专属鉴权流程

## 6. 什么不能进入 `pkg/`

以下内容不应进入 `pkg/`：

1. 服务私有配置模型
2. 领域模型
3. 业务服务实现
4. 服务专属鉴权流程
5. 服务专属仓储实现
6. 只被单个服务使用的内部工具

## 7. 配置建模规则

### 7.1 一服务一配置模型

每个服务的 `config` 包只保留当前服务启动所需字段。

例如，某个服务可以在自己的配置中声明：

- 自己的监听地址
- 自己依赖的下游地址
- 自己使用的数据源地址

这是因为这些字段都属于“当前服务如何启动”的输入，而不是“其他服务的配置模型”。

### 7.2 外部依赖地址仍属于本服务配置

即使某个配置项指向别的服务，例如下游地址、公钥服务地址或管理接口地址，它仍然属于当前服务的启动配置。

因此：

- 依赖别的服务地址，并不等于当前服务拥有了“别的服务的配置模型”

## 8. 引用规则

### 8.1 服务入口引用规则

服务入口应：

1. 从本服务 `internal/config` 读取配置
2. 从 `pkg/` 引入共享基础设施能力

### 8.2 服务内部引用规则

服务内部代码可以直接引用：

- `<workspace-name>/pkg/logging`
- `<workspace-name>/pkg/mysql`
- `<workspace-name>/pkg/otel`

但不应为了“统一风格”而新增一层无意义包装目录。

## 9. 新增共享包前的判断规则

在把代码抽到 `pkg/` 之前，至少应回答以下问题：

1. 它是否真的被多个服务使用。
2. 它是否不携带业务语义。
3. 它是否不依赖服务私有配置模型。
4. 它是否不会反向迫使服务暴露内部实现。

只要其中任一答案不成立，就不应进入 `pkg/`。

## 10. 禁止事项

必须禁止以下做法：

1. 把服务私有配置抽成共享 `config` 包
2. 在 `pkg/` 中放入领域逻辑
3. 在 `pkg/` 中放入服务专属鉴权流程
4. 在某个服务的 `config` 包中声明别的服务配置类型
5. 为了去重而牺牲边界清晰度

## 11. 与其他规范的关系

- 仓库模块与目录分层，可配合《内规-仓库模块与目录约定》使用。
- 共享契约与生成规则，可配合《内规-共享契约与代码生成约定》使用。
- 服务边界与接口所有权，可配合《通用-服务边界与接口所有权》使用。

## 12. 一句话结论

**服务配置永远属于服务自己，`pkg/` 只放无业务语义的共享基础设施能力；边界优先于去重。**
