# 数据访问最小权限通用模板

## 1. 目标与适用范围

本文档定义一套可跨项目复用的数据访问最小权限模板，适用于所有存在“服务拥有自己的数据，并可能被其他服务间接访问”场景的系统。

本文档重点约束四类问题：

1. 数据边界应如何归属。
2. 服务应如何访问自己和他人的数据。
3. 运行时账号应如何收敛最小权限。
4. 跨服务管理动作应如何落到显式接口，而不是直连数据存储。

本文档适用于但不限于以下数据形态：

- 关系型数据库
- 键值存储
- 搜索索引
- 缓存
- 对象存储元数据

本文档不规定具体数据库类型、具体账号系统或具体部署方式。它只回答一个基础问题：

**服务可以调用别的服务获取数据，但不能因为拿得到连接串就直接改别人的数据。**

## 2. 角色定义

为避免文档绑定具体项目，统一使用以下抽象角色：

| 角色 | 定义 |
| --- | --- |
| `数据拥有方服务` | 拥有某份数据模型、数据语义和数据修改规则的服务 |
| `数据访问方服务` | 需要读取或触发修改该数据，但并不拥有该数据语义的服务 |
| `共享基础设施组件` | 提供数据库实例、连接池、备份、监控等基础能力的组件 |
| `管理面接口` | 数据拥有方服务对外暴露的内部管理入口，用于执行受控的跨服务管理动作 |
| `运行时账号` | 服务在日常运行时使用的数据访问身份 |

说明：

- 同一个服务既可能是某类数据的 `数据拥有方服务`，也可能是其他数据的 `数据访问方服务`。
- `共享基础设施组件` 负责承载存储能力，但不拥有业务数据语义。

## 3. 最小权限基本原则

### 3.1 一服务一数据边界

每份业务数据都应有明确拥有方服务。

该服务负责：

1. 定义数据模型。
2. 定义数据写入规则。
3. 定义数据读取语义。
4. 定义外部可见的管理与查询入口。

除拥有方外，其他服务不应直接把这份数据当作自己的本地数据来操作。

### 3.2 运行时权限默认最小化

服务的运行时账号只应拥有完成当前职责所必需的最小权限。

这意味着：

1. 能只读，就不给读写。
2. 能只访问本服务数据，就不给跨服务数据访问。
3. 能不持有 DDL 权限，就不持有。
4. 能通过内部接口完成管理动作，就不通过数据库直连完成。

### 3.3 管理动作与业务读写分离

业务查询、业务写入、内部管理、初始化建库和运维操作应当分层治理，而不是全部复用同一组高权限账号。

否则很容易产生以下问题：

- 日常流量意外获得高权限。
- 某个服务可以跨边界修改其他服务数据。
- 审计时无法分辨正常业务操作和管理操作。

## 4. 数据所有权原则

### 4.1 语义所有权

谁拥有业务，谁拥有该业务对应的数据语义。

这意味着：

1. 拥有方服务定义字段含义。
2. 拥有方服务定义允许哪些状态变更。
3. 拥有方服务定义哪些变更需要额外校验。

调用方服务不应越过拥有方，直接对底层表、集合或索引做语义写入。

### 4.2 修改所有权

数据的写入和状态变更，应当优先由拥有方服务自己执行。

跨服务触发修改时，标准方式应为：

1. 数据访问方调用拥有方服务的内部接口。
2. 拥有方服务执行校验、审计和写入。
3. 拥有方服务返回明确结果。

### 4.3 查询所有权

数据查询也应优先通过拥有方服务暴露的接口完成。

只有在极少数明确治理的共享读模型场景下，才应允许例外读取。即使允许，也必须明确：

1. 读模型来源。
2. 更新延迟。
3. 一致性预期。
4. 权限边界。

## 5. 标准访问模型

### 5.1 服务内访问

服务应直接访问自己拥有的数据，并在本服务内完成：

1. 数据校验。
2. 读写逻辑。
3. 状态变更。
4. 审计记录。

### 5.2 跨服务查询

服务需要其他服务的数据时，应通过被调用方公开的内部查询接口获取，而不是直接读取其底层存储。

这样做的收益：

- 保持数据语义一致。
- 保持审计边界清晰。
- 避免调用方依赖拥有方内部表结构。

### 5.3 跨服务管理动作

服务需要触发其他服务的管理动作时，应通过显式管理面接口完成，而不是直接写对方数据库。

管理面接口应：

1. 权限单独校验。
2. 语义单独定义。
3. 审计单独记录。
4. 与公开业务接口分开治理。

## 6. 账号与权限模型

### 6.1 运行时账号

运行时账号推荐遵循以下原则：

1. 只访问当前服务拥有的数据。
2. 只持有当前服务必要权限。
3. 不复用跨服务高权限身份。

### 6.2 管理账号

管理账号与运行时账号应分离。

典型管理动作包括：

- 建库建表
- 索引变更
- 初始化脚本
- 运维修复

这些动作不应默认由日常业务进程长期持有。

### 6.3 权限分层

推荐至少区分以下权限层级：

1. 只读
2. 读写
3. 管理
4. DDL / 运维

如果系统规模较小，实际账号数量可以少于层级数量，但权限语义仍应明确区分。

## 7. 禁止事项

必须禁止以下做法：

1. 服务运行时直接读写其他服务私有数据。
2. 入口服务直接修改下游领域表。
3. 多个服务共用同一组高权限运行时账号。
4. 将临时排障账号长期用于正式运行流量。
5. 通过数据库直写绕过拥有方服务的业务校验。

## 8. 例外与迁移规则

在遗留系统或迁移阶段，可能存在短期例外，但必须满足以下条件：

1. 例外原因明确。
2. 风险范围明确。
3. 生效期限明确。
4. 回收路径明确。

允许存在迁移桥接，但不允许把桥接方案固化成长期默认模式。

## 9. 落地检查清单

新增服务时，建议至少检查：

1. 该服务拥有的数据边界是否清晰。
2. 运行时账号是否只覆盖本服务必要权限。
3. 是否存在对其他服务数据的隐式直连依赖。

新增内部管理动作时，建议至少检查：

1. 是否由拥有方服务提供管理面接口。
2. 是否有单独权限校验。
3. 是否有单独审计与日志记录。

执行权限收敛或拆库时，建议至少检查：

1. 是否移除了跨服务直库路径。
2. 是否替换为显式内部接口。
3. 是否保留了必要的回归验证与兜底策略。

## 10. 与其他规范的关系

- 服务与接口边界，可配合《服务边界与接口所有权通用模板》使用。
- 服务间认证、授权和用户上下文透传，可配合《服务间通信与权限验证约定》使用。
- 可观测性、缓存刷新和热更新策略，可配合《可观测性与热更新通用模板》使用。

## 11. 一句话结论

**数据属于拥有它的服务，跨服务访问必须走显式接口，运行时权限必须默认收敛到最小。**
