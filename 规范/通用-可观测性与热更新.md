# 可观测性与热更新通用模板

## 1. 目标与适用范围

本文档定义一套可跨项目复用的可观测性与热更新模板，适用于所有存在以下任一场景的系统：

- 跨服务请求调用链
- 运行时缓存
- 配置热更新
- 密钥热更新
- 授权策略热更新

本文档重点约束四类问题：

1. 请求链路应如何被日志与追踪共同证明。
2. 动态状态应如何被缓存、刷新和回退。
3. 热更新行为应如何做到可见、可控和可恢复。
4. 故障发生时应如何保留证据链并执行兜底。

本文档不规定具体观测产品、具体日志库或具体追踪 SDK。它只回答一个基础问题：

**凡是会影响请求结果的动态状态，都必须同时具备可观测证据、受控刷新路径和失败兜底策略。**

## 2. 角色定义

为避免文档绑定具体项目，统一使用以下抽象角色：

| 角色 | 定义 |
| --- | --- |
| `请求入口` | 接收请求并创建根级观测上下文的组件 |
| `调用链节点` | 在请求链路中处理或转发请求的任意服务节点 |
| `配置来源` | 为服务提供配置、密钥、策略或其他动态状态的来源 |
| `缓存持有方` | 在本地缓存动态状态并负责刷新与回退的服务 |
| `观测消费方` | 使用日志、Trace、指标或审计记录定位问题的对象 |

说明：

- 同一个服务既可能是 `调用链节点`，也可能是 `缓存持有方`。
- `配置来源` 可以是数据库、配置中心、公钥服务、策略服务或注册中心。

## 3. 可观测性基本原则

### 3.1 请求可追踪原则

所有关键请求都应具备可追踪能力，至少应支持：

1. 从入口创建根级追踪上下文。
2. 向下游透传追踪上下文。
3. 在每个调用链节点记录本节点处理证据。

### 3.2 日志与链路可关联原则

日志不应是孤立文本，而应当能够与同一次请求的 Trace 对齐。

因此建议：

1. 同一请求在不同服务中的日志能通过统一标识串联。
2. 关键错误日志能定位到具体节点、具体阶段和具体结果。
3. 日志和 Trace 至少有一个稳定关联键。

### 3.3 错误必须留证原则

任何影响用户结果、内部调用结果或动态状态切换结果的错误，都必须留下明确观测证据。

不允许出现以下状态：

- 错误发生了，但没有日志。
- 热更新失败了，但没有记录。
- 缓存刷新了，但无法知道是否命中、是否失败、是否回退。

### 3.4 热更新必须可见原则

凡是会动态改变请求处理结果的状态，如配置、密钥、策略、路由表等，都必须能够在日志或追踪中体现其刷新、切换和失败行为。

## 4. 日志规范模板

### 4.1 最小公共字段

建议每条结构化日志至少带以下字段：

- `ts`
- `level`
- `service`
- `msg`

### 4.2 请求关联字段

在可用时，建议附带：

- `trace_id`
- `span_id`
- `request_id`
- `path`
- `method`
- `target_service`
- `target_method`

### 4.3 安全与权限相关字段

在认证、授权、上下文透传场景下，建议附带：

- `caller_service`
- `auth_result`
- `authz_result`
- `user_id`
- `tenant_id`
- `key_id`

### 4.4 热更新相关字段

在缓存刷新和热更新场景下，建议附带：

- `cache`
- `refresh_result`
- `config_version`
- `key_version`
- `policy_version`
- `source`

### 4.5 禁止事项

日志中不应直接输出以下敏感内容：

- 私钥
- 原始访问令牌
- 完整密码或密钥材料
- 未脱敏的高敏用户信息

## 5. 链路追踪模板

### 5.1 根 Span

请求入口应负责创建根 Span，并将其作为整条链路的起点。

### 5.2 下游透传

调用下游时，应透传追踪上下文，使下游请求能挂接到同一条链路上。

### 5.3 节点内分段

建议在服务内部按关键阶段拆分子 Span 或事件，例如：

1. 认证 / 验签
2. 授权 / 策略判断
3. 缓存读取 / 刷新
4. 数据查询 / 写入
5. 外部依赖调用

### 5.4 错误标记

当请求失败、权限拒绝、验签失败、回源失败或刷新失败时，应在相应 Span 上标记错误状态，并附带可定位的错误信息。

## 6. 缓存与刷新模板

### 6.1 缓存定义要求

任何运行时缓存都应明确以下信息：

1. 数据源是什么。
2. 缓存键是什么。
3. 缓存值承载什么语义。
4. TTL 或有效期如何定义。
5. 刷新触发方式是什么。

### 6.2 刷新策略

缓存刷新通常可采用以下方式之一：

1. 定时轮询。
2. 事件驱动失效。
3. 读取失败后按需刷新。

可以组合使用，但必须明确优先级和最大重试次数。

### 6.3 刷新失败处理

当刷新失败时，应明确以下行为：

1. 是否允许继续使用旧缓存。
2. 旧缓存允许使用多久。
3. 没有旧缓存时是否直接失败。
4. 是否允许单次重试。

### 6.4 禁止事项

不允许每个请求都强制远程回源刷新。

因为这样会：

- 放大依赖方不可用风险。
- 抹平缓存本身的价值。
- 让临时网络抖动直接变成业务故障。

## 7. 热更新模板

### 7.1 允许热更新的对象

常见允许热更新的对象包括：

- 密钥
- 授权策略
- 配置项
- 路由规则
- 可缓存的元数据

### 7.2 触发方式

热更新触发方式应明确为以下一种或多种：

1. 定时轮询。
2. 事件通知。
3. 验证失败后的按需刷新。
4. 管理操作显式触发。

### 7.3 原子切换要求

热更新后的生效切换应尽量原子完成，避免出现长时间半新半旧状态。

至少应满足：

1. 切换前旧值仍可用。
2. 切换成功后新值立即成为主值。
3. 切换失败时不会破坏旧值可用性。

### 7.4 生效时延要求

热更新应明确最大生效时延，避免“最终会生效”但无法验证。

### 7.5 可观测要求

热更新至少应输出以下观测信号之一：

1. 成功切换日志。
2. 失败日志。
3. Trace 事件或子 Span。
4. 状态指标变化。

## 8. 故障处理模板

### 8.1 无缓存且远程不可用

如果本地没有可用缓存，且远程来源不可用，应返回明确错误，而不是静默降级为不可信结果。

### 8.2 有缓存但刷新失败

如果本地仍有未过期缓存，且刷新失败，可在安全边界允许范围内继续使用旧值，并记录刷新失败证据。

### 8.3 刷新后仍失败

如果验证失败、刷新一次后仍失败，应直接按失败处理，不应无限重试。

### 8.4 新旧版本并存

在热更新窗口中，如存在新旧版本短暂并存，应明确：

1. 谁是主值。
2. 旧值允许保留多久。
3. 什么条件下删除旧值。

## 9. 落地检查清单

新增缓存时，建议至少检查：

1. 是否定义了数据源、键、TTL 和刷新策略。
2. 是否定义了失败回退与旧值使用策略。
3. 是否有缓存命中、未命中和刷新日志。

新增热更新能力时，建议至少检查：

1. 是否定义了触发方式和最大生效时延。
2. 是否具备原子切换或等价安全切换方案。
3. 是否具备成功、失败和回退证据。

新增调用链节点时，建议至少检查：

1. 是否创建或继承追踪上下文。
2. 是否能通过日志与 trace 串联。
3. 是否在关键阶段保留足够观测信息。

## 10. 与其他规范的关系

- 服务和接口边界，可配合《服务边界与接口所有权通用模板》使用。
- 数据最小权限和跨服务数据访问，可配合《数据访问最小权限通用模板》使用。
- 服务间认证、授权和用户上下文透传，可配合《服务间通信与权限验证约定》使用。

## 11. 一句话结论

**凡是会影响请求结果的动态状态，都必须同时具备可观测证据、受控刷新路径和失败兜底策略。**
