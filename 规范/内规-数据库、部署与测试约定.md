# 数据库、部署与测试约定

## 1. 目标与适用范围

本文档定义当前仓库在数据库边界、部署组织和测试分层上的现行内规。

本文档适用于当前与未来所有 `services/<service>/` 模块。

本文档重点约束以下问题：

1. 服务如何持有和使用数据库权限。
2. 部署资源和服务实现如何分层。
3. Docker 构建和工作区如何协同。
4. 测试如何按服务内、共享层和跨服务层分层组织。

## 2. 数据库边界约定

### 2.1 一服务一数据边界

每个服务应拥有自己的数据边界。

在关系型数据库场景下，推荐至少做到：

1. 一服务一 schema，或等价的一服务一逻辑数据边界
2. 一服务一组运行时访问凭据
3. 服务默认只访问自己拥有的数据

### 2.2 运行时 DSN 最小化

每个服务在运行时只应注入自己真正需要的 DSN。

不允许出现以下情况：

1. 服务持有其他服务私有库的运行时 DSN
2. 服务持有并不用于正常请求路径的高权限管理 DSN
3. 服务通过“临时方便”的方式直接改写其他服务数据

### 2.3 跨服务管理动作走接口

如果某个服务需要触发另一个服务的数据管理动作，应通过该服务暴露的内部接口完成，而不是直接连库写表。

## 3. MySQL 字符集与排序规则约定

当前仓库中，MySQL 相关配置统一约束为：

1. 字符集：`utf8mb4`
2. 排序规则：`utf8mb4_0900_ai_ci`

### 3.1 实例级约定

MySQL 启动参数必须显式声明：

1. `--character-set-server=utf8mb4`
2. `--collation-server=utf8mb4_0900_ai_ci`
3. `--init-connect=SET NAMES utf8mb4`

### 3.2 连接级约定

运行时 DSN 必须显式包含：

- `charset=utf8mb4`

### 3.3 库表级约定

初始化 SQL 中应显式声明：

1. schema 级字符集与排序规则
2. 表级 `DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci`

不依赖隐式继承作为唯一保障。

## 4. 部署资源组织约定

### 4.1 `ops/` 是联调与部署资源入口

当前仓库中，多服务联调与部署辅助资源统一收敛到 `ops/`，包括：

1. `ops/docker-compose.yml`
2. `ops/mysql/`
3. `ops/otel/`
4. `ops/tests/`

服务目录不应重复维护同一套联调入口。

### 4.2 服务实现与联调资源分离

服务实现留在 `services/<service>/`，联调资源留在 `ops/`。

这样做的目的：

1. 进入服务目录时，思维模型是单服务开发。
2. 进入 `ops/` 时，思维模型是多服务联调和验收。

## 5. Docker 构建约定

### 5.1 构建上下文

当前仓库的服务镜像构建应使用仓库根目录作为 build context。

原因：

1. 需要读取 `go.work`
2. 需要读取 `contracts/`
3. 需要在工作区内解析多模块依赖

### 5.2 Dockerfile 输入约定

服务 Dockerfile 应显式复制工作区和共享契约所需内容，例如：

1. `go.work`
2. `contracts/go.mod`
3. `contracts/`

避免镜像构建时因上下文缺失而无法解析共享模块。

## 6. 测试分层约定

### 6.1 服务内测试

以下测试应放在各自服务目录内：

1. 单元测试
2. 服务私有集成测试
3. 与本服务路由、拦截器、仓储、业务逻辑直接相关的测试

### 6.2 共享模块测试

以下测试应放在共享模块内：

1. `contracts` 契约测试
2. `pkg` 共享基础设施测试

### 6.3 跨服务测试

跨服务测试统一放在 `ops/tests/`，包括：

1. 部署配置校验
2. Docker / Compose 约束校验
3. E2E 测试

跨服务测试不应依赖某个服务的内部实现细节。

## 7. 验证入口约定

当前仓库至少存在以下验证入口：

1. 各服务目录内的 `go test ./...`
2. `contracts` 模块内的生成与测试
3. `pkg` 模块内的共享能力测试
4. `ops/tests` 内的部署校验与 E2E

新增服务后，也应沿用相同结构：

1. 服务内测试归服务自己
2. 共享测试归共享模块
3. 跨服务回归归 `ops/tests`

## 8. 数据变更与重建约定

当初始化 SQL、字符集、排序规则或种子数据发生变化时，应明确是否需要重建数据卷或重新初始化数据库。

对于会影响初始化结果的变更，不应假设已有实例会自动收敛到新状态。

## 9. 禁止事项

必须禁止以下做法：

1. 在运行时为服务注入其他服务私有库的 DSN
2. 用跨服务直库代替内部管理接口
3. 让服务目录和 `ops/` 同时维护重复联调入口
4. 使用服务子目录作为镜像 build context，导致工作区或共享契约缺失
5. 把跨服务测试塞回某个服务目录，导致边界混乱

## 10. 与其他规范的关系

- 数据最小权限原则，可配合《通用-数据访问最小权限》使用。
- 仓库分层和目录职责，可配合《内规-仓库模块与目录约定》使用。
- 共享契约与生成规则，可配合《内规-共享契约与代码生成约定》使用。

## 11. 一句话结论

**数据库权限按服务边界最小化，联调资源统一收敛到 `ops/`，服务镜像依赖仓库根工作区构建，测试按服务内、共享层和跨服务层三层拆分。**
